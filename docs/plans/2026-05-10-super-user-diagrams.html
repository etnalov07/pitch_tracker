1. Role hierarchy

Five tiers. Super User is external (env-var allowlist). The rest live in the DB.

SUPER USER env-var allowlist · SUPER_ADMIN_EMAILS
God mode across all organizations. Read everything; mutate the five targeted actions (force-verify, resend verification, set registration_type, regenerate narrative, soft-delete game).
▼
ORG OWNER organization_members.role = 'owner'
Everything Org Admin can do, plus the self-destruct set:
  • add or remove another admin
  • transfer ownership
  • soft-delete the organization
  • rename the organization
▼
ORG ADMIN organization_members.role = 'admin'
Full local power within this org:
  • create / rename / soft-delete teams
  • invite / remove coaches, assistants, players on any team
  • approve or deny join requests
  • regenerate narratives, force-verify emails of org members
  • soft-delete games on org teams
▼
TEAM COACH / OWNER team_members.role ∈ ('owner','coach','assistant')
Standard team operations on this team only: chart games, manage roster, send invites (gated by upcoming Part C1 fix). Scope ends at the team boundary.
▼
PLAYER team_members.role = 'player'
Personal stats only — own pitching line, own batting line, team scoreboard. No teammate breakdowns, no scouting reports, no charting tools.

2. Data ownership tree

Two co-existing top-level shapes: org-rooted teams (new) and orphaned teams (today's solo coaches).

  • SUPER USER — env-var, no DB row
    • sees everything via /bt-api/admin/*
  • organizations
    • organization_members (role: owner | admin | coach)
      • → users (registration_type='org_admin')
    • teams (organization_id FK)
      • team_members (role: owner | coach | assistant | player)
        • → users (registration_type='coach' or 'player')
      • players (roster; players.user_id nullable)
      • games
        • at_bats → pitches
        • performance_summaries
        • narratives
  • teams (organization_id IS NULL — orphaned, today's behavior)
    • … same children as above …
  • admin_audit ← writes from both super-user and org-level privileged mutations
    • actor_role: 'super' | 'org_owner' | 'org_admin'
    • organization_id: NULL for super-user actions

5. Step-by-step middleware walkthrough

The same request, traced linearly. Use this as the implementation checklist for requireGameAuthority.

  1. Parse and verify the JWT authenticateToken decodes the bearer token, loads req.user = { id, email, ... }. If missing or invalid, return 401 and stop.
  2. Branch on route Super-user route → step 3. Standard game route → step 4.
  3. requireSuperAdmin (super route only) Read SUPER_ADMIN_EMAILS at boot; check req.user.email against the list. Fail → 403. Pass → set req.authority = 'super', jump to step 8.
  4. Load the game and its team Query games by :id, join teams to retrieve organization_id (nullable). If the game doesn't exist, 404.
  5. Try team-scoped authority Look up the caller's team_members row for this team. If role ∈ ('owner','coach','assistant'), set req.authority = 'team' and proceed to step 8.
  6. Try org-scoped authority (only if team is org-linked) Skip if organization_id IS NULL. Otherwise, look up the caller's organization_members row. If role = 'owner' → req.authority = 'org_owner'. If role = 'admin' → req.authority = 'org_admin'.
  7. Deny if nothing matched No team-membership match and no org-membership match → 403. Player and outsider callers terminate here.
  8. Controller dispatch gamesController.softDelete (or adminController.softDeleteGame for the super route) calls the service layer.
  9. Transactional mutation + audit write Inside one DB transaction: UPDATE games SET deleted_at = NOW() WHERE id = :id, then INSERT INTO admin_audit (actor_user_id, actor_role, organization_id, action, target_table, target_id, payload). Audit row is skipped when req.authority = 'team' (open question, see note).
  10. Respond 200 { id, deleted_at }. Done.
Open: the plan's admin_audit.actor_role CHECK constraint allows only 'super' | 'org_owner' | 'org_admin'. So team-coach soft-deletes either go un-audited or we widen the enum to 'team'. The diagram assumes un-audited (matches the plan as written).
