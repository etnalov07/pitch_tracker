## Executive summary

Five biggest wins, ranked by impact-per-effort:

1. **`UX-LG-01` Result-only edit / fix-last-pitch path** *(S1, M, both)* — Single biggest mid-inning frustration source. Today a mis-tapped result requires a 6–8-tap Undo + full re-entry. Add a "Fix Last Pitch" affordance that lets the coach change only the result without re-entering type/zone/velocity. Falls out of a new `PATCH /pitches/:id` endpoint + a Snackbar-style UI.
2. **`UX-PC-01` Consolidate pitch-calling into one implementation** *(S1, L, mobile)* — Two parallel flows (`/live` inline + dedicated `/pitch-calling`) maintain duplicate state, components, and behavior. Pick one, deprecate the other. Will also reveal whether the navy/amber aesthetic in `/pitch-calling` should propagate (Phase 2 question).
3. **`UX-LG-04` Replace native `Alert.alert` with a Snackbar/Toast pattern** *(S2, S, mobile)* — Recurs across Undo, Dropped-K3, End Game, Delete Game, validation errors. Each `Alert.alert` is modal, blocks the inning, breaks design language. Building a single `useToast()` hook lights up ~10 individual findings (`UX-LG-04, -05, -06, -19, UX-PB-08, UX-BP-03, UX-NG-05, UX-DB-02`, and more).
4. **`UX-LG-13` Show current pitcher's pitch count on the live screen header** *(S1, S, mobile)* — Coaches need this for pitch-count rules; today it lives in the tablet stats panel as small body text and is missing from phone entirely. Cheap, high-impact glanceability win.
5. **`UX-NG-02` Smart defaults from last game on New Game setup** *(S1, S, both)* — 80% of games reuse the same opponent / lineup / mode / innings. Pre-fill from the most recent game and let the coach confirm in one tap. Unblocks the "too many setup screens" complaint without redesigning either flow.

Five honorable mentions worth flagging at exec level:

- **`UX-TD-01` Web has no entry point to Pitcher Tendencies modal** *(S1, S, web)* — dead-code path; shipped feature is invisible to web coaches.
- **`UX-IP-01` / `UX-PB-01` Unescapable modals** *(S1, S each)* — `InningChangeModal`, `TeamAtBatModal` have `onDismiss={() => {}}` no-ops. Coach can get stuck.
- **`UX-LG-12` `is_home_game` toggle disappears after first pitch logged** *(S1, M)* — no recovery from a wrong-side setup.
- **`UX-PB-02` Pitcher selector parity gap** *(S1, S)* — web filters by position='P'; mobile shows full roster. Pick one.
- **`UX-GLB-01` Design token consolidation** *(Phase 2, L)* — every screen has hardcoded color/spacing literals. Light-mode-only. Foundation for any future dark-mode or rebrand work.

## Per-screen findings

### Live Game — mobile (`packages/mobile/app/game/[id]/live.tsx`)

**Size:** 2647 lines in one file. Worth flagging on its own (`UX-LG-00` below).

**Current flow — log an on-call strike (mobile phone, our_pitcher mode, pitch calling enabled):**

1. Tap pitch type chip (`PitchTypeGrid`, top)
2. Tap target zone on `StrikeZone` (1st tap — sets `targetZone`)
3. Tap actual location on `StrikeZone` (2nd tap — sets `pitchLocation`)
4. Tap **SEND** (speaks TTS call) — optional but the call-result analytic only fires when active
5. (Optional) Type velocity if no radar
6. Tap result button (`ResultButtons`) → auto-logs

Tap count: **4–6** for an on-call pitch (4 if no call/no velocity, 6 if all features on).

**Current flow — fix a mis-tapped result:**

1. Tap **Undo**
2. Native `Alert.alert` confirmation appears with type+result+count summary
3. Tap "Undo"
4. Re-enter pitch type
5. Re-tap target zone
6. Re-tap actual location
7. (Re-enter velocity if used)
8. Tap correct result

Tap count to fix one wrong tap: **6–8.**

#### Issues

- **`UX-LG-00` · Single screen is 2647 lines** — not a direct UX issue but it makes every future UX change slower and riskier. Split into a `useLiveGameController` hook + 3–4 layout components. *S3, M, mobile*
- **`UX-LG-01` · Result mis-tap requires full re-entry of pitch type + zone + velocity** — Undo deletes the row; no result-only edit path. This is the biggest mid-inning frustration source per the discarded design doc, and the audit confirms it's real. *S1, M, mobile (web parity needed)*
- **`UX-LG-02` · Two-tap-on-zone semantics are not visible** — `StrikeZone` accepts "1st tap = target, 2nd tap = actual location" but the UI gives no cue about which tap the coach is on. For the common case (pitch landed on target), the coach must tap the same zone twice. Easy to mistakenly log target as actual or vice versa. *S1, S, mobile (likely web too)*
- **`UX-LG-03` · No `actual = target` shortcut** — when pitch lands where called, force the coach to tap twice. Should be a single-tap with implicit "actual = target" + a long-press or modifier for off-call entry. *S1, S, mobile + web*
- **`UX-LG-04` · Undo flow uses native `Alert.alert`** — modal, blocking, breaks design language, slow during a fast inning. Replace with snackbar-style "Undo" with 5s timeout (Gmail pattern). *S2, S, mobile*
- **`UX-LG-05` · 3rd-strike-dropped prompt uses native `Alert.alert`** — same problem; this fires inside the per-pitch hot path, so it's worse. *S2, S, mobile*
- **`UX-LG-06` · End Game uses native `Alert.alert`** — acceptable for destructive confirmation but inconsistent with the rest of the (Paper Portal) modal stack. *S3, S, mobile*
- **`UX-LG-07` · Phone layout puts `StrikeZone` deep in a scroll** — vertical order on phone: header → banners → at-bat controls → `PitchTypeGrid` → `StrikeZone` → Send → Shake → Velocity → ResultButtons → Undo → Previous At-Bats. The two biggest tap targets (zone + result) are far apart, requiring scroll between every pitch. *S1, M, mobile*
- **`UX-LG-08` · No "pending pitch summary"** — once type + zone selected, nothing on tablet shows "FB / 1-1 / pending result". On phone only the SEND button does. Coach loses track if they look away. *S2, S, both*
- **`UX-LG-09` · No result-only edit path even via Undo** — see UX-LG-01. The current Undo also wipes the pitch call result attached to the `pitch_calls` row, costing analytic data for what was a UI mistake. *S1, M, both*
- **`UX-LG-10` · Velocity input pulls up the system keyboard mid-inning** — destroys the tap rhythm; the keyboard covers the ResultButtons row. Mitigated by radar but radar isn't ubiquitous. Consider a stepper or chip-based common-velocity picker. *S2, M, mobile*
- **`UX-LG-11` · Pitcher / Hitter tendency buttons are `compact`** — the highest-value mid-inning scouting info is reachable but visually de-prioritized vs. the big colored "SEND" button. Buttons say just "Pitcher" / "Hitter" with no peek at the data. *S2, M, mobile*
- **`UX-LG-12` · Toggle home/away disappears after first pitch logged** — `onSwapPress` is `undefined` once `game.total_pitches > 0`. If the coach realizes the game was set up wrong (wrong home/away, wrong is_home_game) after a few pitches, there's no recovery path inside the live screen. *S1, M, both*
- **`UX-LG-13` · No visible pitch count for current pitcher** — `Total Pitches: N` is in the tablet stats panel (small body text) but not in the phone header at all. CLAUDE.md confirms pitch count "lives in a different surface." Coaches need it on the in-game screen for pitch-count rules. *S1, S, mobile (phone), S, web*
- **`UX-LG-14` · `Hold to Talk` button visibility is conditional** — appears only inside the SEND-call row (when type+zone selected, no active call) or inside the call badge (when call active). If the coach wants to talk to the catcher without sending a call (mound visit, sign confusion), there's no PTT in any other state. *S2, S, mobile*
- **`UX-LG-15` · Shake button semantics opaque** — `pendingShakeCount` increments per tap, passed to the next `speakPitchCall`. UI shows just a badge count; no explanation of what the next call says ("shake 2, then FB / 1-1"). *S3, S, mobile*
- **`UX-LG-16` · "Opp. Pitcher" button label truncates** — uses `currentOpposingPitcher.pitcher_name.split(' ').pop()` (last name only), inside a `compact` button that flexes 1. Long surnames will overflow. *S3, S, mobile*
- **`UX-LG-17` · No `PitchType` smart ordering in opp_pitcher mode** — when no arsenal configured, the PitchTypeGrid shows all 12 types in static order. After the first inning the coach knows what the pitcher actually throws; sort by observed frequency this game. *S2, M, both*
- **`UX-LG-18` · Hardcoded color literals throughout (`#0A1628`, `#F5A623`, `#fef3c7`, `#92400e`, `#b91c1c`, `#6366f1`, `#EF4444`)** — these bypass the Paper theme. Will look broken in dark mode and complicate any future theme refresh. The selectPrompt `#fef3c7`/`#92400e` warning banner in particular won't work in dark mode at all. *S2, M, mobile (and globally — see UX-GLB)*
- **`UX-LG-19` · Selecting a new batter mid-at-bat preserves the current at-bat but creates a UI surprise** — `handleSelectBatter` updates `currentBatter` without showing the new batter is in mid-AB. The displayed name updates but the count + outs continue from the prior batter. Confusing if the coach was trying to *restart* the at-bat. *S2, S, mobile*
- **`UX-LG-20` · `OpposingPitcherModal` and `PitcherStatsModal` overlap functionally** — coach has two modal entry points for "pitcher info" (one for opponent, one for own pitcher stats), but the choice flips based on `gameMode`. The button label is the same shape. Audit visual to confirm. *S3, S, both*

**Size:** `LiveGame.tsx` is 1046 lines; uses `useLiveGameState` + `useLiveGameActions` hooks (good — mobile should follow this split). Layout is two columns: left panel (stats), main panel (top bar + game header + players row + pitch form).

#### Issues

- **`UX-LG-21` · Step labels are present on web but missing for the strike zone** — "Step 1: Select Pitch Type", "Step 3: Velocity", "Step 4: Result". Where's Step 2? It's the strike zone, but unlabeled. Renumber or kill the step labels (they may feel patronizing for repeat coaches anyway). *S3, S, web*
- **`UX-LG-22` · Result-only edit not available on web either** — same as `UX-LG-01`. Web `UndoPitchModal` confirms then deletes; no "change result" path. *S1, M, web (paired with UX-LG-01)*
- **`UX-LG-23` · TopBar action buttons mix emoji styles** — `📋 Scouting`, `🪧 Batter Breakdown`, `🎯 Hitter`, `⚙ Settings`, `⇄ Home/Away`, `↶ Undo`. - **`UX-BP-11` · Pitch type lists are independently hardcoded.** Mobile bullpen `PITCH_TYPE_LABELS` and web bullpen `ALL_PITCH_TYPES` are duplicates of each other AND of the live-game `PitchTypeGrid` constants. Lift into shared. *S3, S, shared package*
- **`UX-BP-12` · Hardcoded color literals on both sides** — mobile bullpen has `#334e68`, `#d9e2ec`, `#15803d`, `#C62828`, `#9ca3af`, `#d1d5db`. Bundle with the global token effort. *S2 — bundle*
- **`UX-BP-13` · No "Undo last pitch" in bullpen** — Live Game has Undo, bullpen doesn't. Same fat-finger risk; equally important during a 30-pitch session. *S2, S, both*
- **`UX-BP-14` · `BullpenPitch.pitch_result` mapping has a load-bearing fallback** — both web and mobile cast bullpen pitch result to `'called_strike'` when null, so the StrikeZone preview colors lie about strike rate when result isn't recorded. Should display these as a neutral color or "unscored" until command grade is computed. *S2, M, both*

### Dashboard entry path (mobile + web)

Mobile: `app/(tabs)/index.tsx` (425 ln). Web: `pages/Dashboard/Dashboard.tsx` (19 ln, role router) → `CoachDashboard.tsx` (445 ln, 450 ln styles).

**Path from launch to charting a pitch:**

- *In-progress game*: tap card → `/live` → start tapping. **2 navigation taps.**
- *Scheduled game*: tap "Start Game" → `/live`. **2 taps.**
- *New game*: tap FAB → `/game/new` → fill form → submit → `/my-lineup` → ... → `/live`. **5+ screens.** (covered by `UX-NG-02`)

#### Issues

- **`UX-DB-01` · Delete game is hidden behind a 400ms long-press** with no visible affordance. Coaches who misclicked or want to clean up a test game can't find delete. Add a swipe-to-delete or per-card kebab menu. *S2, S, mobile*
- **`UX-DB-02` · Delete uses `Alert.alert`** — same global anti-pattern. *S2 — bundled*
- **`UX-DB-03` · No "Resume Last" / "Continue" hero affordance.** If an active game exists it's in the "Active Games" section, but a coach reopening the app between innings would benefit from a one-tap resume at the very top of the screen (or a persistent banner). *S3, S, mobile*
- **`UX-DB-04` · In-progress bullpen sessions are invisible on the dashboard.** Only games are shown. Coach who walks away from a bullpen mid-session has to navigate teams → bullpen → resume. *S2, M, mobile (and web — verify)*
- **`UX-DB-05` · "Stats Cards" duplicate information that the section headers already convey** (Games count = sum of Active + Scheduled + Completed sections). Could trim to recover vertical space. *S3, S, mobile*
- **`UX-DB-06` · "Find a Team" Quick Action is buried mid-dashboard** between Stats and Recent Games. For a new user just joining, this should be more prominent. For an existing coach, it's noise. *S3, S, mobile — surface conditionally*
- **`UX-DB-07` · Status chip color uses `theme.colors.tertiary` for completed games** but tertiary is rarely used elsewhere — visually distinct from the rest of the app. Pick a semantic color (gray/muted). *S3, S, mobile*
- **`UX-DB-08` · Section visual hierarchy is weak.** All section headers use the same `titleMedium` weight with no dividers; sections blur into each other. *S3, S, mobile*
- **`UX-DB-09` · `CoachDashboard.tsx` is 445 lines + 450 lines of styles** — comparable to mobile in scope. Worth a parity pass to verify both surfaces show the same data, sections, and CTAs in equivalent positions. *S3, M, both — code health audit*
- **`UX-DB-10` · Web has a `RoleRouter` (Dashboard.tsx 19 lines routes to Coach/Player/Org dashboard); mobile has only one tab dashboard regardless of role.** Mobile player-role users see a coach-shaped UI. *S2, M, mobile — confirm with super-user-and-signup-modes plan*

## Open questions

Things the audit cannot answer from code alone. Each will reshape the Phase 1 / 2 picks if answered differently.

1. **Should PitchChart adopt the navy/amber scoreboard aesthetic that already exists in `/pitch-calling`?** Three options: (a) yes, lift to tokens and propagate everywhere — Phase 2 theme H above; (b) no, refactor `/pitch-calling` back to the neutral palette; (c) keep as-is and tolerate inconsistency. Affects whether to load Oswald/Source Sans/JetBrains Mono fonts (`UX-GLB-04`).
2. **Real-coach pain ranking** — I scored on principle (heuristic violations) not usage data. Which findings actually pinch the most? If you have a recent support-ticket or coach-feedback snapshot, ranking against that would re-prioritize Phase 1.
3. **Pitch_call ↔ pitch link priority** — `UX-PC-05` proposes a `pitch_calls.pitch_id` FK + 6-bucket result enum. Is the call-execution analytic (per-pitcher, per-game) a real product priority? If yes, this jumps to Phase 1. If "nice to have", it stays Phase 2.
4. **Bullpen Command Grade integration status** — `UX-BP-02` flags that bullpen doesn't show result classification. Per `docs/plans/2026-05-11-zone-based-accuracy.md` and friends, Command Grade is computed — but is it *visible* after each logged pitch? Quick check needed; finding may be already-shipped or genuinely missing.
5. **iOS 26.2 beta status** — per CLAUDE.md, a number of expo libraries are blocked. Phase 2 items (Snackbar lib, `@gorhom/bottom-sheet`, font loading) need to verify old-arch compatibility. Is the beta about to ship stable? If yes, the bans may relax soon and re-open options.
6. **Mobile + web setup paradigm** — `UX-NG-01` flags a fundamental divergence. Wizard or single form? My read is single-form-with-smart-defaults wins for repeat coaches; wizard is friendlier for first-timers. Pick or commit to both.
7. **Scoreboard team-column convention** — `UX-LG-26` flags that opponent is always left, your team always right (regardless of `is_home_game`). MLB convention is home-right. Pick: hold the line on opponent-left (intentional, document), or align with convention (visual swap, possible coach retraining).
8. **`/pitch-calling` route — deprecate or rebrand?** `UX-PC-08`. If the WebSocket relay (catcher-tablet → pitcher-device) is a real use case, this screen has a purpose as the catcher's view. Otherwise, it's dead.
9. **`SituationalCallsRow` — wire up or delete?** `UX-TD-02`. Component is built (PO/BNT/1&3/SHK buttons with colors) but referenced from nowhere. Was it abandoned mid-feature or scoped for a later milestone? Check against `docs/plans/tendencies-buttons-situational-calls.md`.
10. **Web heat zones useful?** `UX-LG-29`. Mobile doesn't have a heat zone overlay. If web users find it useful, lift to mobile. If unused on web too, kill.
11. **What's missing from the audit?** I covered Phase 1 surfaces (live, calling, modals, setup, bullpen, dashboard). Out-of-scope: post-game replay, performance summary, history, scouting report editor, team management, settings, admin. If pain lives there, audit Phase 2 covers them.
