Top 5 highest impact-per-effort

  1. UX-LG-01 Result-only edit / Fix-Last-Pitch path S1 M · Biggest mid-inning frustration source. Today a mis-tapped result requires a 6–8-tap Undo + full re-entry. Add a "Fix Last Pitch" affordance that lets the coach change only the result without re-entering type/zone/velocity.
  2. UX-PC-01 Consolidate pitch-calling into one implementation S1 L · Two parallel flows (/live inline + dedicated /pitch-calling) maintain duplicate state, components, and behavior. Pick one, deprecate the other.
  3. UX-LG-04 Replace native Alert.alert with a Snackbar/Toast pattern S2 S · Recurs across Undo, Dropped-K3, End Game, Delete Game, validation errors. One useToast() hook lights up ~10 findings.
  4. UX-LG-13 Show current pitcher's pitch count on the live screen header S1 S · Coaches need this for pitch-count rules; missing from phone entirely. Cheap, high-impact glanceability win.
  5. UX-NG-02 Smart defaults from last game on New Game setup S1 S · 80% of games reuse the same opponent / lineup / mode / innings. Pre-fill and let the coach confirm in one tap.

Honorable mentions worth exec-level attention

  • UX-TD-01 Web has no entry point to Pitcher Tendencies modal — dead-code path; shipped feature is invisible to web coaches.
  • UX-IP-01 / UX-PB-01 Unescapable modals — InningChangeModal, TeamAtBatModal have onDismiss={() => {}} no-ops. Coach can get stuck.
  • UX-LG-12 is_home_game toggle disappears after first pitch logged — no recovery from a wrong-side setup.
  • UX-PB-02 Pitcher selector parity gap — web filters by position='P'; mobile shows full roster. Pick one.
  • UX-GLB-01 Design token consolidation — every screen has hardcoded color literals. Light-mode-only. Foundation for any future dark-mode or rebrand work.

Per-screen findings

Live Game — mobile (packages/mobile/app/game/[id]/live.tsx) 33 findings

Size: 2647 lines in one file. Worth flagging on its own (UX-LG-00).

Current flow — log an on-call strike (phone, our_pitcher mode):
1. Tap pitch type chip · 2. Tap target zone (1st tap) · 3. Tap actual location (2nd tap) · 4. Tap SEND (TTS) · 5. Type velocity · 6. Tap result → auto-logs
Tap count: 4–6 per pitch.
Current flow — fix a mis-tapped result:
1. Tap Undo · 2. Confirm in Alert.alert · 3–8. Re-enter pitch type + target + actual + (velocity) + correct result
Tap count: 6–8 to fix one wrong tap.

Pitch Calling — mobile (app/game/[id]/pitch-calling.tsx + BT/TTS) 12 findings
Current flow — send + log a call (dedicated screen):
1. Tap pitch type · 2. Tap zone · 3. Tap SEND (TTS speaks "<Pitch>, <Zone>... <Pitch>, <Zone>" over A2DP) · 4. Tap result chip (strike / ball / foul / in-play)

Stack reality: Calls are spoken via expo-speech TTS, routed via react-native-audio-api (iOS A2DP) / expo-av (Android). Double-speak repeats once for crowd resilience (~3–4s). Walkie-talkie uses separate walkieTalkie.ts mic passthrough.

Tight bundle that together meaningfully improves daily-use UX without touching the design system. Estimated effort: ~1.5–2 sprints for one engineer.

UX-LG-04 / UX-GLB-02 Replace Alert.alert with Snackbar · S2/S+M
Cross-cutting; one component lights up ~10 findings. Pre-requisite for several other items. Single visible pattern improvement.
2
UX-LG-01 / UX-LG-22 Fix Last Pitch (result-only edit) · S1/M
Biggest mid-inning friction source. Needs new PATCH /pitches/:id (result-only) endpoint + Snackbar "Edit" affordance after each pitch.

UX-LG-13 Pitch count in live header · S1/S
Coaches need this for pitch-count rules. Trivial layout addition; immediate glanceability win.

UX-LG-12 is_home_game recoverable after first pitch · S1/M

UX-NG-02 Smart defaults from last game · S1/S

6
UX-PB-01 + UX-IP-01 Make all modals escapable · S1/S each
Fix onDismiss={() => {}} no-ops in TeamAtBatModal and InningChangeModal. Two-line fixes; eliminate "stuck modal" reports.
7
UX-TD-01 Wire up Pitcher Tendencies button on web · S1/S

8
UX-PB-02 Pitcher selector position filter parity · S1/S

Open questions

Things the audit cannot answer from code alone. Each will reshape Phase 1 / 2 picks if answered differently.

  1. Should PitchChart adopt the navy/amber scoreboard aesthetic from /pitch-calling? Three options: (a) yes, lift to tokens and propagate; (b) no, refactor /pitch-calling back to neutral; (c) keep as-is and tolerate inconsistency. Affects whether to load Oswald/Source Sans/JetBrains Mono fonts.
  2. Real-coach pain ranking — I scored on principle, not usage data. Recent support-ticket or coach-feedback snapshot would re-prioritize Phase 1.
  3. Pitch_call ↔ pitch link priority — UX-PC-05 proposes a pitch_calls.pitch_id FK + 6-bucket result enum. Is the call-execution analytic a real product priority?
  4. Bullpen Command Grade integration status — UX-BP-02 flags bullpen doesn't show result classification. Is Command Grade visible after each logged pitch, or computed-but-hidden?
  5. iOS 26.2 beta status — CLAUDE.md bans several expo libs. Phase 2 (Snackbar lib, bottom-sheet, fonts) needs to verify old-arch compat. Stable shipping soon?
  6. Mobile + web setup paradigm — UX-NG-01 flags a fundamental divergence. Wizard or single form? My read: single-form-with-smart-defaults wins for repeat coaches.
  7. Scoreboard team-column convention — UX-LG-26. Opponent always left, your team always right (regardless of is_home_game). MLB convention is home-right. Hold the line or align?
  8. /pitch-calling route — deprecate or rebrand? UX-PC-08. If the WebSocket relay is a real use case, screen has a purpose as catcher's view. Otherwise dead.
  9. SituationalCallsRow — wire up or delete? UX-TD-02. Component built (PO/BNT/1&3/SHK) but referenced from nowhere. Abandoned mid-feature or scoped for later?
  10. Web heat zones useful? UX-LG-29. Mobile doesn't have it. If useful on web, lift to mobile. If unused, kill.
  11. What's missing from the audit? Phase 1 surfaces only. Out-of-scope: replay, performance summary, history, scouting editor, team mgmt, settings, admin. If pain lives there, Phase 2 covers them.
