## Plan (Decisions)

### `UX-NG-08` + `UX-NG-09` · Mobile date/time pickers (item #10)

Free-text `YYYY-MM-DD` / `HH:MM` `<TextInput>`s are bad on touch — typos, no calendar widget, no time-zone safety.

**Decision:** add `@react-native-community/datetimepicker` (not on the iOS 26.2 beta blocklist; doesn't use TurboModules so works under the no-new-arch constraint). Replace the two TextInputs with `<Pressable>` buttons that open the native picker. State internally is a single `Date` (gameDateTime) instead of two strings — eliminates the parse step in `handleCreate`.

Platform-specific UI:
- iOS: `display="inline"` for date, `display="spinner"` for time. Both render in-page with a "Done" button to close.
- Android: `display="default"` — native dialog that closes on selection.

### `UX-LG-02` + `UX-LG-03` · Zone-tap UX cleanup (item #9)

Today the StrikeZone takes two taps per pitch (target zone → actual location). No on-screen cue tells the coach which tap is being captured, and there's no shortcut for the common case (pitch landed on the called target).

**Decision:** keep the existing two-tap behavior intact; layer two affordances on top inside `live.tsx`:

1. **Zone-tap hint subtitle** above the StrikeZone — phase-aware text:
   - No target picked: "**Tap target zone** · 1st tap"
   - Target picked, no actual: "**Tap actual location** · 2nd tap (or pin to target below)"
   - Both set: "✓ Target + actual set — tap result to log" (green)
2. **"Pitch hit target (zone label)"** outlined button between StrikeZone and the call/result rows, visible only when target is set but actual isn't. Tap fills `pitchLocation` with the target zone's center coords (using `PITCH_CALL_ZONE_COORDS`). One-tap shortcut for on-spot pitches.

No changes to the StrikeZone component itself — keeps the shared component stable (used by bullpen + breakdown viewer too).

### `UX-LG-12` · `is_home_game` recovery (item #4)

Today the swap button (`onSwapPress` / web `SwapButton`) disappears once `game.total_pitches > 0`. No recovery path if the coach realizes the game was set up wrong after a few pitches.

**Decision:** enable the swap post-pitches with a confirmation dialog. Server-side `gameService.toggleHomeAway` is a clean one-field flip (just `UPDATE games SET is_home_game = NOT ...`); it doesn't touch scores, lineups, or pitches. The scoring convention in this codebase keeps opponent runs in `away_score` and user runs in `home_score` regardless of `is_home_game`, so scores don't move when the flag flips — only the inning-half batting context changes for future pitches.

Confirmation message explains exactly what's about to happen: "X pitches already logged. Switching makes your team the {away/home} team going forward — already-logged pitches keep their inning-half. Scores stay attached to each team."

Applied to both mobile and web. Mobile uses the existing `useConfirm` hook landed in `2026-05-23-snackbar-foundation.md`; web adds `useConfirm` to `useLiveGameActions`.
